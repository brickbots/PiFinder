# Civil datetime on `shared_state` is stored timezone-aware in UTC

`shared_state` carries two unrelated kinds of "time": the **measurement epoch** (a `time.time()` wall clock used for "how old is this fix?" — `estimate_time`, IMU `timestamp`, see [ADR-0013](./0013-pointing-estimate-timing.md)) and the **civil datetime** (the calendar date + clock used as the astronomical epoch that turns RA/Dec into Alt/Az and drives planet/comet ephemerides). This ADR fixes the contract for the second: it is always stored as a timezone-aware UTC `datetime`, normalised at the `set_datetime()` boundary, and read through the explicit `utc_datetime()` / `local_datetime()` accessors.

**Naive vs aware** (Python `datetime`): a **naive** datetime has no zone (`tzinfo is None`, `utcoffset()` is `None`) — it reads `2024-06-28 11:00:00` with no statement of *which* 11:00, so it names no absolute instant. `datetime.now()` and `datetime.strptime(...)` return naive values. An **aware** datetime carries a zone/offset — `2024-06-28 11:00:00+00:00` (UTC) or `…13:00:00+02:00` (Brussels in June) — which pins it to one absolute instant; `13:00+02:00` and `11:00+00:00` are the *same* instant. `datetime.now(pytz.utc)` and `tz.localize(naive)` produce aware values.

## Context

The status screen showed identical "UTC TM" and "LCL TM" values when the clock was set by hand, yet the expected offset appeared once a GPS fix arrived.

- **Root cause:** `set_datetime()` localised only *naive* inputs to UTC; a timezone-aware *local* datetime — which is exactly what the manual time/date-entry UI produces via `timezone.localize(...)` — was stored as-is, in local time. The status screen then formatted it with `datetime().time().isoformat()`, and `.time()` silently drops the tzinfo, printing the local wall clock under the "UTC" label.
- **Astronomical consumers were unaffected.** `radec_to_altaz` and `ts.from_datetime` use the *absolute instant* of an aware datetime, so storing local-vs-UTC never changed the math. Only display paths that discard the zone (`.time()`) were wrong.
- **There was no written contract** for what `shared_state.datetime()` returns, so each of ~25 call sites assumed its own answer. The motivating change (upstream PR #508) also surfaced two bugs rooted in argument-less `datetime.now()` — which returns a **naive** value whose digits are the *host OS's* local timezone (the system clock's configured zone: a third zone, distinct from both UTC and the observer's location zone, and not guaranteed to be UTC on the device):
  - `catalogs.py` did `now().replace(tzinfo=utc)`, which makes the value **aware** by *relabelling* it UTC instead of converting — so an OS-local reading like `13:00` on a UTC+2 box is asserted to be `13:00Z` when the real instant is `11:00Z`. Correct only if the OS runs UTC. Fixed with `datetime.now(pytz.utc)`: **aware** and genuinely UTC.
  - `callbacks.set_time` took *today's date* from the naive OS-local `now()` but combined it with the user's entered clock time in the **target** (location) zone; when the OS-local date and the target-zone date differ around midnight, the result lands a day off. Fixed with `datetime.now(timezone)`: read "now" **aware in the target zone** so its calendar date is the right one.

## Decision

- **Storage is always timezone-aware UTC**, normalised at the `set_datetime()` boundary: a naive input is interpreted as UTC (`pytz.utc.localize`); an aware input is converted (`astimezone(pytz.utc)`). Naive-vs-aware is tested with `utcoffset() is None` (the correct check) rather than `tzname() is None`.
- **Callers MUST pass timezone-aware datetimes.** "Naive ⇒ UTC" is a safety net, not an interface. Every current producer already emits UTC or offset-aware time: real GPS (UBX) → `tzinfo=utc`; gpsd (`convert_datetime`) → aware UTC; manual entry → aware local; catalogs → `now(pytz.utc)`. The only naive inputs are UTC by construction (GPS epoch arithmetic, test fixtures).
- **Read through explicit accessors:** `utc_datetime()` for UTC, `local_datetime()` for the active location's zone (UTC fallback). Bare `datetime()` is retained and, after normalisation, also returns UTC; new code should prefer the explicit accessors so the intended zone is on the page.
- **Civil datetime and measurement epoch are different clocks with different jobs** and must not be conflated. Civil datetime answers "what is the sky doing now" (calendar / ephemeris); measurement epoch answers "how old is this fix" (`time.time()` deltas). See [ADR-0013](./0013-pointing-estimate-timing.md).

## Considered options

- **Make bare `datetime()` the single UTC accessor and drop `utc_datetime()`** — rejected for now: ~25 call sites read `datetime()`. An explicit `utc_datetime()` / `local_datetime()` pair states intent at the call site and lets `datetime()` be deprecated incrementally rather than in one risky sweep.
- **Store local, convert on read** — rejected: that pushes timezone handling into every consumer (the bug we just had) and leaves "the stored instant" ambiguous. Normalising once at the boundary is the single source of truth.
- **Migrate persisted data** — unnecessary: nothing persists a naive-local civil datetime. `shared_state`'s datetime is in-memory only (re-acquired from GPS or manual entry after every boot); observations store an absolute epoch (`local_datetime().timestamp()`); telemetry stores ISO-8601 *with offset*, so even recordings made before this change round-trip correctly.

## Consequences

- After this change `utc_datetime()` is functionally an explicit alias of `datetime()` (storage is already UTC). That is deliberate belt-and-suspenders: the value is the documented call-site intent, not a different result. If `datetime()` is later deprecated, `utc_datetime()` becomes the sole UTC accessor.
- One twin of the original display bug remains, as known debt rather than by decision: `main.py` sets `location.last_gps_lock = datetime().time().isoformat()[:8]` — the same tz-dropping `.time()` pattern, feeding the **GPS LKT** field. It is safe today (written only on a real GPS lock, which is UTC) but should migrate to `utc_datetime()` for consistency.
- Status label **GPS LST → GPS LKT**: "LST" collides with Local Sidereal Time; the field is the *last GPS lock time*, so "LKT" (lock-time). A vocabulary fix, not a behaviour change.
- A regression test guards `set_datetime` (`tests/test_state_datetime.py`): (a) naive input → UTC; (b) aware-local input → the same absolute instant in UTC (asserted on bare `datetime()`, so a storage regression fails); (c) UTC TM ≠ LCL TM for a non-UTC location. The bug class ("fine until DST or a non-UTC box") is exactly what silent timezone handling hides.
- **Enforcement.** `PiFinder/timez.py` is the single sanctioned place to build datetimes (`utc_now` / `local_now` / `utc_from_timestamp` / `parse` / `naive` / `utc`), and ruff's `DTZ` (flake8-datetimez) family is enabled repo-wide with `timez.py` as the only exemption. A bare `datetime.now()` / `datetime(...)` / `strptime()` / `fromtimestamp()` elsewhere now fails lint, so "local or UTC?" is an explicit choice at every call site and a naive value cannot silently reach the civil/astronomy path. The migration also fixed a latent relabel bug in the comet-freshness check (`comets.py` read a file mtime as naive-local and stamped it UTC; it now uses the true UTC instant).
